4 Principios de llaves: identificadores únicos y verificación de consistencia

Antes de unir cualquier par de tablas, es fundamental verificar que las llaves (las variables que conectan ambas tablas) sean consistentes. Una llave suele ser un identificador único como folioviv, foliohog o el nombre de un país.

4.1 ¿Por qué importa verificar las llaves?

Unir tablas sin verificar las llaves puede producir:

  • Filas duplicadas, si una llave aparece más de una vez en la tabla de la derecha.
  • Pérdida de datos, si hay valores en una tabla que no tienen pareja en la otra.

4.2 Verificar unicidad con distinct() y nrow()

Podemos comprobar si una variable es un identificador único comparando el número de filas totales con el número de valores distintos:

# ¿Cada vivienda aparece solo una vez en la tabla de viviendas?
nrow(viviendas)
nrow(distinct(viviendas, folioviv))

Si ambos números son iguales, folioviv es una llave única en viviendas. Si difieren, hay duplicados y debemos resolverlo antes de unir.

4.3 Detectar duplicados con count()

viviendas %>%
  count(folioviv) %>%
  filter(n > 1)

Si esta tabla está vacía, no hay duplicados y podemos proceder con confianza.

5 Uniones/fusión de tablas

Las uniones (o joins) permiten combinar información de dos tablas a partir de una o más variables llave en común. En {dplyr} existen cuatro tipos principales.

5.1 left_join(): conservar toda la tabla izquierda

left_join() mantiene todas las filas de la tabla izquierda y agrega las columnas de la tabla derecha cuando hay coincidencia en la llave. Si no hay coincidencia, las columnas de la derecha se rellenan con NA.

Vamos a agregar las características de las viviendas (derecha) a los hogares (izquierda.)

base_hog_viv <- hogares %>%
  left_join(viviendas, by = "folioviv")

nrow(hogares)       # Filas originales
nrow(base_hog_viv)  # Filas después del join (deben ser iguales)

Podemos encadenar uniones. Ahora, vamos a añadir los datos de hogares y viviendas a la tabla de personas:

base_completa <- poblacion %>%
  left_join(hogares,   by = c("folioviv", "foliohog")) %>%
  left_join(viviendas, by = "folioviv")
Nota¿Por qué dos llaves?

Al unir poblacion con hogares usamos c("folioviv", "foliohog") porque una vivienda puede tener varios hogares. Solo la combinación de ambas columnas identifica un hogar de forma única.

5.2 inner_join(): solo filas con coincidencia en ambas tablas

inner_join() devuelve únicamente las filas que tienen coincidencia en ambas tablas. Las filas sin pareja se descartan.

Por ejemplo, hogares que sí tienen registros de gastos en consultas médicas.

base_gasto_hog <- gastoshogar %>%
  inner_join(hogares, by = c("folioviv", "foliohog"))

nrow(gastoshogar)    # Registros de gasto
nrow(base_gasto_hog) # Solo los que tienen hogar en la tabla principal

5.3 full_join(): conservar todo de ambas tablas

full_join() mantiene todas las filas de ambas tablas. Donde no haya coincidencia, los valores faltantes se rellenan con NA.

base_full <- hogares %>%
  full_join(viviendas, by = "folioviv")

5.4 anti_join(): diagnóstico de no coincidencias

anti_join() devuelve las filas de la tabla izquierda que no tienen pareja en la tabla derecha. Es una herramienta de diagnóstico.

¿Qué hogares tienen gasto médico pero NO están en la tabla de hogares?

anti_join(gastoshogar, hogares, by = c("folioviv", "foliohog"))

¿Qué hogares NO tienen ningún registro de gasto médico?

anti_join(hogares, gastoshogar, by = c("folioviv", "foliohog")) %>%
  nrow()
Importanteanti_join() como verificación previa

Es buena práctica correr anti_join() antes del join. Si devuelve muchas filas inesperadas, revisa el formato de las llaves (mayúsculas, espacios, tipos de datos).

5.5 Unir filas con bind_rows()

Cuando dos tablas tienen la misma estructura de columnas y queremos apilarlas verticalmente (por ejemplo, datos de distintos trimestres), usamos bind_rows():

trim1 <- ENSU_CB_0325 %>%
  select(ID_VIV, ID_PER, CVE_ENT, NOM_ENT, SEXO, EDAD, starts_with("BP1_2")) %>%
  mutate(trimestre = "Primero")

trim2 <- ENSU_CB_0625 %>%
  select(ID_VIV, ID_PER, CVE_ENT, NOM_ENT, SEXO, EDAD, starts_with("BP1_2")) %>%
  mutate(trimestre = "Segundo")

trim3 <- ENSU_CB_0925 %>%
  select(ID_VIV, ID_PER, CVE_ENT, NOM_ENT, SEXO, EDAD, starts_with("BP1_2")) %>%
  mutate(trimestre = "Tercero")

trim4 <- ENSU_CB_1225 %>%
  select(ID_VIV, ID_PER, CVE_ENT, NOM_ENT, SEXO, EDAD, starts_with("BP1_2")) %>%
  mutate(trimestre = "Cuarto")

ensu_2025 <- bind_rows(trim1, trim2, trim3, trim4)

6 De muchos a uno

¿Cómo tendría que hacer si a la base de hogares, quisiera pegarle la información de la edad de sus integrantes, pero quedarme con la unidad de análisis del hogar?

Tenemos que agregar la edad (que está a nivel persona) a nivel hogar. Podemos calcular un promedio, por ejemplo.

poblacion2 <- poblacion %>% 
  select(folioviv, foliohog, numren, edad) %>%     #seleccionamos sólo las que nos interesan
  group_by(folioviv, foliohog) %>%  
  #generamos la agrupación
  summarise(promedio_edad = round(mean(edad), 0)) #calculamos el promedio y redondeamos a cero decimales

Ahora unimos:

agregada <- poblacion2 %>% 
  full_join(hogares, by = c("folioviv", "foliohog"))

7 Reestructuración: pivot_longer() y pivot_wider()

Los datos pueden estar en dos formatos: ancho (wide), donde cada variable ocupa una columna, y largo (long), donde una columna identifica la variable y otra contiene el valor.

7.1 De ancho a largo: pivot_longer()

7.1.1 Ejemplo: Índice de Desarrollo Humano

La base idh tiene una columna por año. Para graficar la evolución temporal o calcular estadísticas por año, necesitamos el formato largo.

Primero verificamos la estructura:

glimpse(idh)

Antes de reestructurar, nos aseguramos de que todas las columnas de años sean numéricas:

idh <- idh %>%
  mutate(across(3:10, ~ round(as.numeric(.x), 3)))

Ahora aplicamos pivot_longer():

idh_largo <- idh %>%
  pivot_longer(
    cols      = 3:10,        # Columnas a reestructurar
    names_to  = "annio",     # Nombre de la nueva columna de identificador
    values_to = "idh_valor"  # Nombre de la nueva columna de valores
  ) %>%
  mutate(idh_valor = as.numeric(idh_valor))

glimpse(idh_largo)
Nota¿Qué hizo pivot_longer()?

Tomó 8 columnas (una por año) y las convirtió en dos columnas: annio (que contiene el nombre del año) e idh_valor (que contiene el puntaje). El número de filas aumentó proporcionalmente.

7.2 De largo a ancho: pivot_wider()

pivot_wider() es la operación inversa: toma una columna de identificadores y una de valores, y crea una columna separada para cada identificador.

7.2.1 Ejemplo: Restaurar el IDH a formato ancho

idh_ancho <- idh_largo %>%
  pivot_wider(
    names_from  = annio,     # Los valores únicos de esta columna se vuelven nombres de columna
    values_from = idh_valor  # Los valores de esta columna llenan las nuevas columnas
  )

glimpse(idh_ancho)
TipResumen: ¿cuándo usar cada función?
Función Cuándo usarla
pivot_longer() Para análisis, visualización, group_by() + summarise()
pivot_wider() Para presentar resultados en tablas o restaurar el formato original

## Reestructuración: `pivot_longer()` y `pivot_wider()`

Los datos pueden estar en dos formatos: **ancho** (*wide*), donde cada variable ocupa una columna, y **largo** (*long*), donde una columna identifica la variable y otra contiene el valor.
